# SkyLensServerless follow-up permissions/embed fix plan

## Scope
- Restore the exported/live `Permissions-Policy` contract so camera and geolocation remain required capabilities alongside accelerometer, gyroscope, and magnetometer.
- Realign the embedded viewer delegation contract and validation copy with the required AR capability set.
- Update tests and parity docs so policy/delegation drift is caught and described accurately.
- Preserve the previously shipped direct CelesTrak path and desktop/mobile viewer behavior; do not broaden viewer logic changes beyond what is needed for contract parity.

## Milestone
### M1. Restore permissions and embed contract parity
- Update the source-of-truth policy string in `SkyLensServerless/next.config.ts` to include `camera=(self)` and `geolocation=(self)` without dropping the existing motion sensor directives.
- Keep `public/_headers` byte-for-byte aligned with the generated static-host header string exported from `next.config.ts`.
- Update `app/embed-validation/page.tsx` so the page copy and iframe `allow` string explicitly cover camera, geolocation, accelerometer, gyroscope, and magnetometer.
- Refresh committed export artifacts generated by `npm run build`, including `out/_headers` and the embed-validation export output that embeds the copied delegation text/string.

## Interface definitions
- `SKYLENS_PERMISSIONS_POLICY`: single source-of-truth response header contract for local Next headers, `public/_headers`, and exported `_headers`; required features are `camera`, `geolocation`, `accelerometer`, `gyroscope`, and `magnetometer`.
- Embedded iframe delegation contract: `app/embed-validation/page.tsx` and exported embed-validation artifacts must expose an `allow` attribute that delegates the same required AR capabilities needed by embedded viewer flows.
- Static export parity contract: preview/runtime checks must continue to treat committed `out/_headers` and `out/embed-validation*.{html,txt}` as generated outputs that reflect the current source contract after build regeneration.

## Validation plan
- Update unit assertions in `SkyLensServerless/tests/unit/next-config.test.ts` so they lock the full five-feature policy string and the parity between `next.config.ts` and `public/_headers`.
- Update `SkyLensServerless/tests/unit/serve-export.test.ts` so exported preview responses still surface the same header contract from `_headers`.
- Update embed assertions in `SkyLensServerless/tests/e2e/embed.spec.ts` and any nearby string assertions so both the response header and iframe `allow` delegation require camera and geolocation in addition to motion sensors.
- Run targeted regression coverage for the viewer flow that must remain intact: permissions/mobile viewer coverage (`tests/e2e/permissions.spec.ts`) plus the existing unit suites that guard mobile viewer shell behavior if touched by fallout.
- Regenerate and verify exported artifacts with `npm run build`; use the build output as the canonical refresh path instead of hand-editing `out/` files where possible.

## Compatibility and rollback
- Compatibility: this is a contract restoration, not a feature removal. Hosts that honor `_headers` should continue working, but they must now expose camera and geolocation again for required AR flows.
- Rollback: revert the policy/delegation contract together across `next.config.ts`, `public/_headers`, embed validation copy, tests, and regenerated `out/` artifacts as one unit to avoid split-brain behavior between local runtime and exported preview.

## Regression notes
- Do not change viewer startup ordering, fallback modes, direct CelesTrak fetching, or mobile overlay layout unless a test proves the permissions contract restoration requires a minimal follow-up adjustment.
- Treat `next.config.ts` as the authoritative contract and avoid introducing a second hand-maintained permissions helper unless implementation shows repeated drift that cannot be eliminated with current exports/constants.
- Because committed `out/` artifacts are part of the checked-in surface, implementation must verify that generated files reflect the updated contract and do not preserve stale embed-validation copy.

## Risk register
| Risk | Why it matters | Control |
| --- | --- | --- |
| Source/header artifact drift | Local Next headers, `public/_headers`, and committed `out/_headers` can silently diverge. | Update the shared policy constant first, assert parity in unit tests, then regenerate `out/` artifacts from build. |
| Embedded contract remains incomplete | An iframe `allow` list missing camera/geolocation would leave embedded AR flows inconsistent with product requirements. | Update page copy and `allow` together; keep e2e coverage on both the response header and iframe attribute. |
| Regressing prior viewer/mobile fixes | Touching permissions-related surfaces can accidentally disturb the verified live viewer startup and mobile overlay behavior. | Keep viewer logic changes out of scope, and rerun the existing permissions/mobile regression coverage after contract updates. |
| Stale docs/parity guidance | PARITY text currently documents the reduced contract, which can mislead future changes. | Update PARITY/docs in the same slice as code/test changes so future edits follow the restored contract. |
